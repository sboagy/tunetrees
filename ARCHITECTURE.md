# TuneTrees Architecture Guide

This is the top-level architecture guide for AI agents working in the TuneTrees repo.

TuneTrees is organized around an offline-first runtime and a schema-driven build pipeline. Postgres is introspected by oosync, and that generated contract keeps the browser SQLite layer, shared metadata, and Cloudflare Worker aligned.

## Repo Topology

TuneTrees is developed alongside a related repo:

- `tunetrees`: the main app repo. This is where the SolidJS UI, local SQLite wrapper, generated artifacts, worker entrypoint, tests, and app-specific sync overrides live.
- `oosync`: the schema-agnostic sync tooling repo. This repo owns the generic code generator, shared sync contracts, sync runtime, and generic worker implementation.

Important consequence:

- TuneTrees consumes oosync as a dependency via `node_modules/oosync`, even when the adjacent `oosync` repo is open for reference or development.
- App-specific behavior belongs in TuneTrees config, generated artifacts, SQL comments, or app wiring. It should not be hard-coded into oosync unless the behavior becomes truly reusable and schema-agnostic.

## Mental Model

Use this model when reasoning about changes:

1. Postgres is the authoritative remote schema.
2. `oosync/src/codegen-schema.ts` introspects that schema and generates the artifacts TuneTrees consumes.
3. The browser app runs entirely against local SQLite WASM through Drizzle.
4. SQLite triggers populate the outbox (`sync_push_queue`) when local rows change.
5. oosync's runtime pushes local changes to the worker and pulls remote changes back down.
6. The worker applies generated sync policy against Postgres and returns normalized changes.

The UI should feel local-first and immediate. Sync is a background transport and reconciliation system, not the source of truth for the live screen.

TuneTrees operates as one node within a larger multi-repo workspace. It relies on a shared core library (`rhizome`) and a generated sync runtime (`oosync`).

## What Is Generated vs Hand-Written

Generated artifacts are a core part of the architecture, not build noise.

Generated by `npm run codegen:schema`:

- `drizzle/schema-sqlite.generated.ts`: generated SQLite Drizzle schema derived from Postgres.
- `drizzle/schema-sqlite.ts`: generated SQLite schema wrapper used by the browser runtime.
- `shared/generated/sync/table-meta.generated.ts`: generated sync contract and table facts shared by app and worker.
- `shared/table-meta.ts`: generated app-facing table metadata projection layered on top of the shared generated contract.
- `src/lib/db/client-sqlite.ts`: generated thin wrapper around oosync's generic browser SQLite runtime.
- `src/lib/sync/runtime-config.ts`: generated thin wrapper that injects the browser SQLite runtime into `@oosync/sync`.
- `worker/src/generated/schema-postgres.generated.ts`: generated Postgres Drizzle schema for the worker.
- `worker/src/generated/worker-config.generated.ts`: generated worker config after defaults and TuneTrees overrides are merged.
- `worker/src/index.ts`: generated sync worker entrypoint that injects TuneTrees artifacts into the generic oosync worker.

Hand-written glue that stays stable around those artifacts:

- `oosync.codegen.config.json`: the main TuneTrees-owned customization point for generation and sync policy overrides.
- `src/lib/db/oosync-browser-sqlite-hooks.ts`: TuneTrees-specific browser DB hooks for views, compatibility objects, and migration-time cleanup.
- `src/lib/db/init-view-column-meta.ts`, `src/lib/db/init-views.ts`, `src/lib/db/migration-version.ts`: app-specific helpers consumed by the browser DB hook layer.
- `worker/src/handler.ts`, `worker/src/media.ts`: app-owned worker wrapper/routes that extend the generated sync worker with authenticated media handling.

Rule: do not hand-edit generated files. If generated output is wrong, fix the Postgres schema, Postgres comments, `oosync.codegen.config.json`, or the generator.

## Data Flow

This architecture has two important flows: artifact flow and runtime flow.

### 1. Artifact Flow: Postgres -> Generated Contract -> App + Worker

1. TuneTrees defines remote schema in Postgres and Supabase SQL.
2. `npm run codegen:schema` runs `node_modules/oosync/src/codegen-schema.ts` from the TuneTrees repo.
3. oosync introspects Postgres catalogs, constraints, types, and table comments.
4. oosync merges those inferred facts with TuneTrees overrides from `oosync.codegen.config.json`.
5. oosync writes the generated SQLite schema wrapper, shared table metadata, app-facing table metadata, browser client/runtime wrappers, worker Postgres schema, worker config, and worker entrypoint into TuneTrees-owned paths.
6. The browser app imports the generated SQLite schema wrapper, generated browser client wrapper, and generated runtime-config wrapper. The worker wrapper imports the generated Postgres schema, worker config, and generated sync worker entrypoint.

In TuneTrees, schema agreement is produced by codegen rather than maintained manually in parallel.

### 2. Runtime Flow: User Action -> Local SQLite -> Worker -> Postgres -> Local SQLite

1. Auth/bootstrap code initializes a per-user sql.js database in the browser through the generated `src/lib/db/client-sqlite.ts` wrapper, which delegates to oosync's generic browser runtime and restores persisted state from IndexedDB.
2. The SolidJS UI reads and writes local SQLite through Drizzle.
3. SQLite triggers write row changes into the local outbox table `sync_push_queue`.
4. `src/lib/db/oosync-browser-sqlite-hooks.ts` supplies TuneTrees-specific browser DB hooks such as view initialization, compatibility objects, and migration-time local cleanup.
5. `src/lib/sync/runtime-config.ts` injects TuneTrees runtime dependencies into oosync via the generated browser runtime wrapper: local schema, syncable tables, table ordering, SQLite instance access, trigger control, outbox backup helpers, persistence, and logging.
6. Offline media uses external browser storage rather than SQLite blobs: `src/lib/media/media-vault.ts` stores pinned audio and queued note-upload drafts in IndexedDB, while a local-only SQLite table (`media_draft_outbox`) tracks pending note-media uploads that still need an authenticated worker round-trip.
7. `AuthContext` starts the oosync sync service with the local DB, auth session, and any request overrides, and also runs offline-media maintenance (draft upload replay plus pinned-audio lookahead prefetch) when the app is online.
8. The browser sends pending changes and pull state to the Cloudflare Worker via `/api/sync`, while authenticated media uploads/downloads go through `/api/media/*`.
9. The worker evaluates generated sync rules, applies push policy against Postgres, pulls matching remote rows, and returns remote changes.
10. oosync applies those remote changes back into local SQLite, suppressing triggers while applying remote data and restoring them afterward.
11. SolidJS reacts to local DB changes. The UI continues to read local state; it does not switch to remote reads after sync.

At runtime, the browser always renders from the local database. Sync only changes what is in that local database.

## Sync Layer

The sync layer is oosync plus TuneTrees-owned artifacts and overrides.

Defining characteristics:

- Schema knowledge is generated from Postgres introspection rather than hand-maintained in app code.
- Worker behavior is defined by generated schema plus declarative config, not by scattering table-specific assumptions across app code.
- The browser sync runtime is generic; TuneTrees-specific behavior is supplied through generated artifacts, runtime wiring, and config.

How TuneTrees uses oosync:

- `@oosync/sync` provides the generic sync runtime used by the browser.
- `src/lib/sync/runtime-config.ts` supplies TuneTrees-specific runtime dependencies.
- `worker/src/index.ts` calls the generic worker factory with TuneTrees-generated artifacts, while `worker/src/handler.ts` is the deployed wrapper entrypoint for app-specific routes.
- Generated table metadata provides syncable tables, primary keys, unique keys, timestamp columns, deleted-flag knowledge, and related core facts.
- Generated worker config provides pull/push rules, ownership collections, sanitization, auth binding, and retry behavior.

The worker is still app-specific at the artifact layer, but not at the implementation layer.

## How `oosync.codegen.config.json` Is Used In TuneTrees

TuneTrees treats `oosync.codegen.config.json` as the consumer-owned overlay on top of Postgres introspection defaults.

### Output Paths

The file pins all generated artifacts to TuneTrees-owned paths:

- `drizzle/schema-sqlite.generated.ts`
- `drizzle/schema-sqlite.ts`
- `shared/generated/sync/table-meta.generated.ts`
- `shared/table-meta.ts`
- `src/lib/db/client-sqlite.ts`
- `src/lib/sync/runtime-config.ts`
- `worker/src/generated/schema-postgres.generated.ts`
- `worker/src/generated/worker-config.generated.ts`
- `worker/src/index.ts`

That layout matters because TuneTrees owns the artifacts even though oosync generates them.

### Browser Runtime Config

TuneTrees also uses the `browserSqlite` config block to drive the generated browser wrappers:

- storage key prefixes and IndexedDB naming
- local schema/database versioning
- SQLite migration file list and ordering
- forced-reset query parameter behavior
- test-only browser hook names
- the hand-written hook module path (`src/lib/db/oosync-browser-sqlite-hooks.ts`)

This keeps the generic browser runtime in oosync while preserving TuneTrees-specific behavior in a small, explicit hook surface.

### Table Metadata Overrides

TuneTrees uses `tableMeta` overrides to keep app-specific semantics out of oosync core:

- `excludeTables`: excludes sync-internal tables such as `sync_change_log`.
- `changeCategoryByTable`: labels tables with TuneTrees concepts like `catalog`, `repertoire`, `practice`, and `user` for app-facing semantics.
- `normalizeDatetimeByTable`: normalizes timestamp-shaped strings for tables like `practice_record` and `daily_practice_queue`.
- `tableToSchemaKeyOverrides`: preserves naming where default snake_case -> camelCase mapping is not enough.

### Worker Policy Overrides

TuneTrees uses `worker.config` to express pull/push behavior declaratively:

- `collections`: defines ownership collections such as `userProfileIds` and `selectedGenres`.
- `pull.tableRules`: declares how tables are filtered for a user. Some tables use simple ownership rules, while others use compound filters or RPC-backed pull rules.
- `push.tableRules`: customizes delete behavior, auth-bound properties, sanitization, and retry payload rules.

Examples in TuneTrees:

- `plugin`, `goal`, and `tune` use declarative pull rules with ownership/public logic.
- `note` and `reference` use RPC pull rules with explicit `paramMap` bindings.
- `practice_record` push behavior is customized with `denyDelete`, minimal retry payloads, and timestamp sanitization.
- `user_profile` and related user-owned tables bind auth user IDs through worker rules instead of ad hoc client logic.

This file is the first place to look when sync behavior is correct in principle but wrong for TuneTrees-specific policy.

## Hard Boundaries & Invariants

These boundaries are strict. AI agents should treat them as design rules, not suggestions.

### 1. SolidJS UI Boundary

- The UI reads and writes the local SQLite database, not Postgres.
- UI code belongs in `src/**` and should use local queries, services, and Drizzle-backed access patterns.
- Do not make the UI depend on worker internals, remote schema types, or app-specific sync rule evaluation.
- Sync is asynchronous infrastructure. The live screen should remain understandable from local DB state alone.

### 2. Generated SQLite Boundary

- `drizzle/schema-sqlite.generated.ts` is generated from Postgres introspection and is the schema foundation for the browser database.
- `drizzle/schema-sqlite.ts`, `src/lib/db/client-sqlite.ts`, and `src/lib/sync/runtime-config.ts` are generated wrapper files.
- TuneTrees-specific browser behavior belongs in `src/lib/db/oosync-browser-sqlite-hooks.ts`, not in generated wrappers.
- Generated schema and metadata files are write-only outputs. Manual edits will be wiped on the next codegen run.
- If schema-derived output is wrong, fix the remote schema, config, or generator. Do not patch generated output by hand.

### 3. Shared Metadata Boundary

- `shared/generated/sync/*` is the pure generated contract shared by app and worker.
- `shared/table-meta.ts` is TuneTrees-owned output for app-facing metadata, but it is still generated.
- oosync core must not import TuneTrees app code. The contract must stay data-only and portable.

### 4. Worker Boundary

- The Cloudflare Worker owns remote Postgres access and sync policy enforcement.
- `worker/src/index.ts` should remain a thin artifact-injection layer around the generic oosync worker.
- Browser code must not bypass the worker for normal sync responsibilities.
- Worker behavior should be expressed through Postgres schema, Postgres comments, generated metadata, and `oosync.codegen.config.json`, not hidden in UI code.

### 5. Multi-Repo Boundary

- TuneTrees-specific table names, ownership rules, and domain semantics belong in TuneTrees inputs and config.
- oosync must stay schema-agnostic. If a change requires hard-coded TuneTrees table logic inside `oosync_arch`, stop and redesign it around codegen or configuration.
- Only move logic into oosync when the abstraction is truly generic and reusable.

### 6. Generated-Artifact Workflow Invariant

- Schema change workflow is: change Postgres schema or comments -> update `oosync.codegen.config.json` if needed -> run `npm run codegen:schema` -> commit regenerated artifacts.
- Drift checking is done with `npm run codegen:schema:check`.
- In TuneTrees, codegen expects a local Postgres/Supabase instance reachable at `127.0.0.1:54322` unless an explicit safe override is provided.

## Key Runtime Components

### Browser / App

- `src/lib/db/client-sqlite.ts`: generated thin wrapper around oosync's generic browser SQLite runtime.
- `src/lib/db/oosync-browser-sqlite-hooks.ts`: TuneTrees-specific hook layer for views, compatibility tables/columns, and migration cleanup.
- `src/lib/db/index.ts`: intentionally exports the browser SQLite boundary only; Postgres access happens in the worker.
- `src/lib/auth/AuthContext.tsx`: initializes the per-user local DB and starts the oosync sync service.
- `src/lib/sync/runtime-config.ts`: generated runtime wiring for `@oosync/sync`.

### Worker

- `worker/src/index.ts`: passes generated schema tables, syncable tables, table registry core, and worker config into the generic worker factory.
- `worker/src/generated/*`: generated worker-facing schema and config derived from Postgres + TuneTrees overrides.

### oosync Repo

- `oosync/src/codegen-schema.ts`: generator entrypoint.
- `oosync/src/runtime/browser-sqlite.ts`: generic browser sql.js runtime and sync runtime factory used by generated TuneTrees wrappers.
- `oosync/src/sync/*`: generic sync runtime used by TuneTrees.
- `oosync/worker/src/*`: generic worker runtime that TuneTrees configures via generated artifacts.

## Offline-First Principles

These principles define the product behavior and should be preserved.

1. Local-first writes: All database writes go to SQLite immediately.
2. Async sync: Changes queue for background upload.
3. Sync-before-download: Upload pending changes before pulling remote.
4. Conflict resolution: Last-write-wins (based on timestamp).
5. Graceful degradation: App fully functional when offline.

## Spaced Repetition (FSRS)

TuneTrees uses the FSRS algorithm (`ts-fsrs`):

```text
User rates tune -> FSRS calculates -> Next review date stored
       |
  Again (1) -> Short interval (1 day)
  Hard  (2) -> Medium interval
  Good  (3) -> Scheduled interval
  Easy  (4) -> Long interval (2x normal)
```

Key tables:

- `practice_record`: immutable historical practice events.
- `repertoire_tune`: current scheduling state per tune within a repertoire.
- `daily_practice_queue`: frozen daily queue snapshot used to present the day's work.

This domain logic is independent of oosync. oosync moves the rows; TuneTrees services decide what those rows mean.

## Guidance For AI Agents

When working in this repo, start by deciding which layer owns the change:

- UI/workflow change: stay in `src/**` and keep the UI local-first.
- Local schema/import stability change: use the stable wrappers around generated artifacts.
- Sync policy or ownership change: inspect `oosync.codegen.config.json` first.
- Remote schema shape change: update Postgres-side schema/comments, then regenerate.
- Generic sync/runtime limitation: verify whether it belongs in `oosync_arch` as a schema-agnostic improvement.
- Browser SQLite lifecycle issue: prefer extending oosync's generic browser runtime or the TuneTrees hook module instead of hand-expanding generated wrappers.
- Do not bypass the generated Drizzle schema to write raw SQL queries in the UI layer. All local data access must flow through `src/lib/db/client-sqlite.ts`.

Do not treat generated files as the place where business logic lives. In TuneTrees, the durable architecture is: remote schema and config as inputs, generated artifacts as contract, local SQLite as runtime source of truth, and oosync as the transport and reconciliation layer.
