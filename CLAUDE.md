# TuneTrees — Claude Code Instructions

Repository-specific architecture, tech stack details, and scoped patterns live in the root `AGENTS.md` and `*/AGENTS.md` files. Read `ARCHITECTURE.md` before touching sync/codegen boundaries, runtime data flow, generated artifacts, or repository responsibilities.

## Tech Stack

- Frontend: SolidJS + TypeScript (Vite)
- Styling/UI: Tailwind + shadcn-solid + `@kobalte/core` primitives
- Data grids: `@tanstack/solid-table` + `@tanstack/solid-virtual`
- Local DB: SQLite WASM via `sql.js` + IndexedDB (Drizzle ORM)
- Remote DB (browser): Supabase JS wrapped by Drizzle (`src/lib/db/client-postgres-browser.ts`)
- Remote: Supabase (Auth + Postgres + Realtime)
- Sync: outbox-driven bidirectional sync via Cloudflare Worker + `oosync`
- Scheduling: `ts-fsrs` (FSRS); Music notation: `abcjs`
- PWA: `vite-plugin-pwa` + Workbox; deploy via Cloudflare Pages
- Lint/format: Biome; Tests: Vitest + Playwright

## Common Commands

```bash
npm run typecheck          # TypeScript check — run after every code change
npm run lint               # Biome lint
npm run format             # Biome format
npm run test               # Unit tests (Vitest)
npm run test:e2e           # E2E tests (Playwright)
npm run codegen:schema     # Regenerate all sync/schema artifacts
npm run codegen:schema:check
npm run db:local:reset     # Reset local Supabase + auth fixtures
```

Before handing off changes, run `npm run format`, `npm run lint`, and `npm run typecheck`.

## Analysis Before Action

- If requirements are ambiguous, ask before writing code.
- Before any edit, read the current file state to confirm symbols and context match your plan.
- Plan tests before implementing. Prefer deterministic tests: one input state, one expected output.
- Check for existing utilities/patterns before inventing new ones.

## Generated Files — Do Not Edit

The following files are generated by `npm run codegen:schema` and **must never be hand-edited**:

- `drizzle/schema-sqlite.generated.ts`
- `drizzle/schema-sqlite.ts`
- `worker/src/generated/schema-postgres.generated.ts`
- `worker/src/generated/worker-config.generated.ts`
- `shared/generated/sync/table-meta.generated.ts`
- `shared/table-meta.ts`
- `src/lib/db/client-sqlite.ts`
- `src/lib/sync/runtime-config.ts`
- `worker/src/index.ts`

Any file whose header says `AUTO-GENERATED`, `GENERATED`, or `DO NOT EDIT` is off-limits. Edit the source input instead and regenerate.

### Schema Change Workflow

1. Apply Supabase migration locally (`npx supabase db push --local` or rhizome equivalent)
2. Run `npm run codegen:schema`
3. Verify: `npm run codegen:schema:check && npm run typecheck && npm run lint && npm run test:unit`

## Type Safety

- TypeScript only; avoid `any`. Prefer `unknown` + narrowing at third-party boundaries.
- Run `npm run typecheck` immediately after any code generation or modification. Do not finish a turn until it passes.

## Reliability

- Stop on error: if an edit causes a syntax error, type error, or failing test, stop, explain the failure, re-read the file from disk, and propose a corrected patch.
- After 2 failed edit attempts on the same file without new information, stop and ask for guidance.
- Never append code that already exists. Check for duplicate imports, exports, helpers, and config blocks before finalizing.

## Process Guardrails

- Never commit, push, or open PRs unless explicitly requested.
- Never commit directly to `main`; prefer a new branch + PR.
- Do not start the dev server unless explicitly requested.
- Do not introduce React patterns (no React hooks, no React-only libraries in app code paths).
- The `legacy/` directory is reference-only; do not modify it for new work.
- If a task changes architecture/sync/codegen boundaries, update `ARCHITECTURE.md` in the same change unless it remains accurate as-is.
